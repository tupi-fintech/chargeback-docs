# Actualización de Notificación de Estado

![Notificaciones del Sistema](../images/system-notifications.png)

## Endpoint Único para Webhooks

Utilizamos un único endpoint de webhook para todos los eventos de la plataforma. El tipo de evento se identifica por el campo `event` en el payload.

Este documento describe específicamente el evento `status`.

## Tupi envía una notificación webhook (HTTP POST) cada vez que el estado de una disputa de contracargo cambia:

```jsonc
{
    "event": "status",
    "payload": {
        "contractDisputeId": "d492bhhnshts7160i5g0", // ID Interno de Tupi
        "acquirerReferenceNumber": "20099984311312200798013", // ARN (puede ser null)
        "transactionIdentifier": "431220079801", // TID (puede ser null)
        "helpdeskCaseIdentifier": "HD-2025-001234", // ID del caso helpdesk (puede ser null)
        "actionStatus": "waiting_for_merchant", // enum
        "workflowStatus": "analysis" // enum
    }
}
```

### Campos de la Notificación Webhook

| Campo | Tipo | Obligatorio | Descripción |
|-------|------|-------------|----------|
| `event` | string | Sí | Identificador del tipo de evento. Siempre "status" para notificaciones de actualización de estado |
| `payload` | object | Sí | Contenedor para los datos de actualización de estado |
| `payload.contractDisputeId` | string | Sí | Identificador único interno de Tupi para el contrato de disputa |
| `payload.acquirerReferenceNumber` | string | No | Número de Referencia del Adquirente (ARN). Puede ser null |
| `payload.transactionIdentifier` | string | No | ID de la Transacción (TID). Puede ser null |
| `payload.helpdeskCaseIdentifier` | string | No | Identificador del caso helpdesk asociado a esta disputa. Puede ser null |
| `payload.actionStatus` | string | Sí | Estado operativo actual que indica quién debe actuar en la disputa |
| `payload.workflowStatus` | string | Sí | Etapa actual del pipeline interno de procesamiento de la disputa |

## Valores de Action Status - `payload.actionStatus`

El campo `actionStatus` indica el estado operativo actual de la disputa y quién es responsable de la próxima acción.

| Valor | Descripción |
|-------|-----------|
| `received_from_issuer_network` | Disputa recibida de la red emisora |
| `waiting_for_merchant` | Esperando que el comercio proporcione una respuesta o documentación |
| `received_from_merchant` | Respuesta recibida del comercio |
| `waiting_for_issuer_network` | Respuesta enviada y esperando a la red emisora |
| `done` | Disputa completamente resuelta |

## Valores de Workflow Status - `payload.workflowStatus`

El campo `workflowStatus` indica la etapa actual de la disputa en el pipeline interno de procesamiento de Tupi.

| Valor | Descripción |
|-------|-----------|
| `draft` | Disputa creada pero aún no procesada |
| `enrichment` | Esperando enriquecimiento de datos de la transacción o del comercio |
| `filtering_automation` | Reglas de filtrado automatizado siendo evaluadas |
| `assignment_automation` | Reglas de asignación automatizada siendo aplicadas |
| `merchant_activation_automation` | Activación automatizada del comercio en progreso |
| `review_automation` | Reglas de revisión automatizada siendo evaluadas |
| `assignment` | Disputa siendo asignada manualmente a un analista |
| `analysis` | Disputa bajo análisis de un analista |
| `review` | Trabajo del analista bajo revisión |
| `execution` | Respuesta de la disputa siendo ejecutada y enviada a la red |
| `done` | Flujo de trabajo completado |
