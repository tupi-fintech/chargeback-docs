# Atualização de Notificação de Status

![Notificações do Sistema](../images/system-notifications.png)

## Endpoint único para webhooks

Utilizamos um único endpoint de webhook para todos os eventos da plataforma. O tipo do evento é identificado pelo campo `event` no payload.

Este documento descreve especificamente o evento `status`.

## A Tupi envia uma notificação webhook (HTTP POST) sempre que o status de uma disputa de chargeback for alterado:

```jsonc
{
    "event": "status",
    "payload": {
        "contractDisputeId": "d492bhhnshts7160i5g0", // ID Interno da Tupi
        "acquirerReferenceNumber": "20099984311312200798013", // ARN (pode ser null)
        "transactionIdentifier": "431220079801", // TID (pode ser null)
        "helpdeskCaseIdentifier": "HD-2025-001234", // ID do caso helpdesk (pode ser null)
        "actionStatus": "waiting_for_merchant", // enum
        "workflowStatus": "analysis" // enum
    }
}
```

### Campos da Notificação Webhook

| Campo | Tipo | Obrigatório | Descrição |
|-------|------|-------------|----------|
| `event` | string | Sim | Identificador do tipo de evento. Sempre "status" para notificações de atualização de status |
| `payload` | object | Sim | Container para os dados de atualização de status |
| `payload.contractDisputeId` | string | Sim | Identificador único interno da Tupi para o contrato de disputa |
| `payload.acquirerReferenceNumber` | string | Não | Número de Referência do Adquirente (ARN). Pode ser null |
| `payload.transactionIdentifier` | string | Não | ID da Transação (TID). Pode ser null |
| `payload.helpdeskCaseIdentifier` | string | Não | Identificador do caso helpdesk associado a esta disputa. Pode ser null |
| `payload.actionStatus` | string | Sim | Status operacional atual indicando quem precisa agir na disputa |
| `payload.workflowStatus` | string | Sim | Estágio atual do pipeline interno de processamento da disputa |

## Valores de Action Status - `payload.actionStatus`

O campo `actionStatus` indica o estado operacional atual da disputa e quem é responsável pela próxima ação.

| Valor | Descrição |
|-------|-----------|
| `received_from_issuer_network` | Disputa recebida da rede emissora |
| `waiting_for_merchant` | Aguardando o estabelecimento fornecer uma resposta ou documentação |
| `received_from_merchant` | Resposta recebida do estabelecimento |
| `waiting_for_issuer_network` | Resposta enviada e aguardando a rede emissora |
| `done` | Disputa totalmente resolvida |

## Valores de Workflow Status - `payload.workflowStatus`

O campo `workflowStatus` indica o estágio atual da disputa no pipeline interno de processamento da Tupi.

| Valor | Descrição |
|-------|-----------|
| `draft` | Disputa criada, mas ainda não processada |
| `enrichment` | Aguardando enriquecimento de dados da transação ou estabelecimento |
| `filtering_automation` | Regras de filtragem automatizada sendo avaliadas |
| `assignment_automation` | Regras de atribuição automatizada sendo aplicadas |
| `merchant_activation_automation` | Ativação automatizada do estabelecimento em andamento |
| `review_automation` | Regras de revisão automatizada sendo avaliadas |
| `assignment` | Disputa sendo atribuída manualmente a um analista |
| `analysis` | Disputa sob análise de um analista |
| `review` | Trabalho do analista sob revisão |
| `execution` | Resposta da disputa sendo executada e enviada à rede |
| `done` | Fluxo de trabalho concluído |
