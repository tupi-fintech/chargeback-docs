# Cycle Update Notification

![System Notifications](../images/system-notifications.png)

## Tupi sends a webhook notification (HTTP POST) whenever a chargeback dispute cycle or step is updated:

```jsonc
{
    "event": "cycle",
    "payload": {
        "contractDisputeId": "d492bhhnshts7160i5g0", // Tupi Internal ID
        "contractDisputeWorkflowStatus": "execution", // enum, apply financial adjustments only when the status is "execution"
        "name": "chargeback", // enum: chargeback | pre_arbitrage | arbitrage
        "status": "open", // enum: open | closed
        "decision": null, // enum: accepted_in_full | accepted_partially | rejected | null
        "steps": [
            {
                "name": "chargeback", // enum
                "status": "open", // enum: open | closed
                "message": "Chargeback received from issuer network",
                "memberInCharge": "acquirer", // enum: acquirer | issuer | scheme
                "acquirerStepAction": "act", // enum: act | wait | done
                "startDate": "2025-01-01T00:00:00.000Z",
                "endDate": null,
                "actionDueDate": "2025-01-15T00:00:00.000Z"
            }
        ],
        "installmentAdjustments": [
            {
                "installmentNumber": 1,
                "acquirerReferenceNumber": "25699994259255249310221", // ARN
                "issuerDispatchDate": "2025-01-01T00:00:00.000Z",
                "acquirerIssuerFinancialSolution": "full_credit", // enum
                "acquirerIssuerAmount": 10000, // in cents = 100.00
                "acquirerDispatchDate": null,
                "acquirerLiability": "merchant_debt", // enum
                "acquirerAmount": 10000, // in cents = 100.00
                "merchantDispatchDate": null,
                "acquirerMerchantFinancialSolution": "debit_merchant", // enum
                "acquirerMerchantAmount": 10000 // in cents = 100.00
            }
        ],
        "recomendation": "Accept the chargeback and refund the cardholder",
        "requestForInformation": false,
        "informationReceived": false,
        "startDate": "2025-01-01T00:00:00.000Z",
        "endDate": null
    }
}
```

### Webhook Notification Fields

| Field | Type | Required | Description |
|-------|------|-------------|----------|
| `event` | string | Yes | Event type identifier. Always "cycle" for cycle update notifications |
| `payload` | object | Yes | Container for cycle update data |
| `payload.contractDisputeId` | string | Yes | Tupi's internal unique identifier for the dispute contract |
| `payload.contractDisputeWorkflowStatus` | string | Yes | Current workflow status of the dispute. Apply financial adjustments only when the status is "execution" |
| `payload.name` | string | Yes | Name of the dispute cycle |
| `payload.status` | string | Yes | Current cycle status. Values: "open", "closed" |
| `payload.decision` | string | No | Final cycle decision when closed. null if still open |
| `payload.steps` | array | Yes | Array of step objects representing each phase within the cycle |
| `payload.installmentAdjustments` | array | No | Array of financial adjustment objects per installment |
| `payload.recomendation` | string | No | Tupi's recommendation regarding the dispute outcome |
| `payload.requestForInformation` | boolean | Yes | Whether additional information was requested from the involved parties |
| `payload.informationReceived` | boolean | Yes | Whether the requested additional information was received |
| `payload.startDate` | string | Yes | Cycle start date in ISO 8601 format |
| `payload.endDate` | string | No | Cycle end date in ISO 8601 format. Null if not yet closed |

### Step Object Fields - `payload.steps[]`

| Field | Type | Required | Description |
|-------|------|-------------|----------|
| `name` | string | Yes | Step identifier within the cycle |
| `status` | string | Yes | Current step status. Values: "open", "closed" |
| `message` | string | No | Descriptive message about this step |
| `memberInCharge` | string | Yes | Network member currently responsible for the next action |
| `acquirerStepAction` | string | Yes | Action required from the acquirer at this step |
| `startDate` | string | Yes | Step start date in ISO 8601 format |
| `endDate` | string | No | Step end date in ISO 8601 format. Null if not closed |
| `actionDueDate` | string | No | Deadline for the required action in ISO 8601 format |

### Installment Adjustment Object Fields - `payload.installmentAdjustments[]`

| Field | Type | Required | Description |
|-------|------|-------------|----------|
| `installmentNumber` | number | Yes | Installment sequence number |
| `acquirerReferenceNumber` | string | Yes | Acquirer Reference Number (ARN) for this installment |
| `issuerDispatchDate` | string | No | Date the issuer dispatched the financial adjustment in ISO 8601 format |
| `acquirerIssuerFinancialSolution` | string | No | Acquirer's financial decision in its relationship with the issuer |
| `acquirerIssuerAmount` | number | No | Adjusted amount at the issuer level in cents (e.g., 10000 = 100.00) |
| `acquirerDispatchDate` | string | No | Date the acquirer dispatched the financial adjustment in ISO 8601 format |
| `acquirerLiability` | string | No | Acquirer's liability classification for the dispute |
| `acquirerAmount` | number | No | Adjusted amount at the acquirer level in cents (e.g., 10000 = 100.00) |
| `merchantDispatchDate` | string | No | Date the merchant's financial adjustment was processed in ISO 8601 format |
| `acquirerMerchantFinancialSolution` | string | No | Acquirer's financial decision for the merchant installment (payout perspective) |
| `acquirerMerchantAmount` | number | No | Adjusted amount at the merchant level in cents (e.g., 10000 = 100.00) |

## Cycle Name Values - `payload.name`

| Value | Description |
|-------|-----------|
| `chargeback` | First cycle — initial dispute raised by the issuer on behalf of the cardholder |
| `pre_arbitrage` | Second cycle — escalated dispute before formal arbitration |
| `arbitrage` | Third cycle — formal arbitration submitted to the card scheme |

## Cycle Decision Values - `payload.decision`

| Value | Description |
|-------|-----------|
| `accepted_in_full` | Full disputed amount accepted (merchant loses) |
| `accepted_partially` | Partial amount of the dispute accepted |
| `rejected` | Dispute rejected — transaction returned to the counterparty |
| `null` | No decision yet — cycle open |

## Step Name Values - `payload.steps[].name`

| Value | Description |
|-------|-----------|
| `chargeback` | Initial chargeback received from the issuer network |
| `chargeback_response` | Acquirer's response to the initial chargeback |
| `pre_arbitrage` | Pre-arbitrage escalation raised by the issuer |
| `pre_arbitrage_response` | Acquirer's response to the pre-arbitrage |
| `arbitrage` | Formal arbitration submitted to the card scheme |
| `arbitrage_response` | Acquirer's response to the arbitration |

## Member In Charge Values - `payload.steps[].memberInCharge`

| Value | Description |
|-------|-----------|
| `acquirer` | The acquirer (Tupi's client) is responsible for the next action |
| `issuer` | The card issuer is responsible for the next action |
| `scheme` | The card scheme (Visa, Mastercard, etc.) is processing the step |

## Acquirer Step Action Values - `payload.steps[].acquirerStepAction`

| Value | Description |
|-------|-----------|
| `act` | Action required (respond, provide documentation, etc.) |
| `wait` | Waiting — another party is responsible for the current step |
| `done` | Actions completed for this step |

## Acquirer Liability Values - `payload.installmentAdjustments[].acquirerLiability`

| Value | Description |
|-------|----------|
| `loss` | Acquirer absorbs the loss. Common cases: processing error, authorization failure |
| `merchant_debt` | Merchant absorbs the loss. Common cases: fraud, commercial dispute |

## Acquirer Merchant Financial Solution Values - `payload.installmentAdjustments[].acquirerMerchantFinancialSolution`

These values represent the acquirer's financial decision for a given installment in its relationship with the merchant.

| Value | Description |
|-------|----------|
| `keep_as_is` | Do nothing — maintain as-is. If scheduled: keep scheduled. If settled: do not recover funds. If canceled: keep canceled |
| `hold_funds` | Temporarily block merchant funds — hold the money with the acquirer. Temporary decision. If scheduled: temporarily blocks payout (debt / suspected amount). If settled: generate temporary block of equivalent value in merchant's future schedule. If canceled: not applicable |
| `cancel_payout` | Payout will be canceled definitively. If scheduled: cancel payout definitively. If settled: not applicable. If canceled: not applicable |
| `update_amount` | Update value in case of partial acceptance. If scheduled: update payout value. If settled: not applicable. If canceled: not applicable |
| `debit_merchant` | Total loss for merchant. If scheduled: not applicable. If settled: generate definitive debt of equivalent value in merchant's future schedule. If canceled: not applicable |

## Acquirer Issuer Financial Solution Values - `payload.installmentAdjustments[].acquirerIssuerFinancialSolution`

These values represent the acquirer's financial decision for a given installment in its relationship with the issuer.

> **Note:** Partial adjustments must be resolved via fee collection.

| Value | Description |
|-------|----------|
| `full_credit` | Credit for the issuer in the full amount of the coupon |
| `accept_chargeback` | Received chargeback accepted (subsequent debit allowed) — no insistency on collection |
| `representment` | Received chargeback contested and collection maintained. General flow: send a new coupon charge given the disagreement with the dispute. Visa Allocation: initiate a pre-arbitrage |
| `keep_as_is` | Do nothing — maintain as-is. If scheduled: keep scheduled. If settled: do nothing. If canceled: keep canceled. If sent: do nothing. If confirmed by issuer: do nothing |
| `cancel_credit` | Cancel sending of coupons scheduled at the acquirer. If scheduled: cancel |

> **Warning:** Tupi's system does not perform any financial decision or adjustment directly in the client's system. The notifications described in this documentation are for informational purposes and reflect decisions made by the analyst. Receiving, interpreting, and processing the events is the exclusive responsibility of the client's system, which must follow its own business rules and internal policies.
